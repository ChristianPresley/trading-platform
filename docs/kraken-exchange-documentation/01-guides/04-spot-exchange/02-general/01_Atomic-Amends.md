# Spot Atomic Amends

> Source: https://docs.kraken.com/api/docs/guides/spot-amends

## Overview

The atomic amend model is a fast, single phase transaction that enables clients to modify order parameters in-place within the trading engine.

**Key characteristics of amended orders:**

- Keep the same order identifiers assigned by Kraken and/or client
- Keep execution information on the order for any previous fills
- Keep queue priority in the order book where possible

**Example Request:**

```json
{
    "nonce": 1695828490,
    "cl_ord_id": "6d1b345e-2821-40e2-ad83-4ecb18a06876",
    "order_qty": "1.25",
    "limit_price": "690795"
}
```

## Background

### Legacy Edit Transactions

The legacy `editOrder` endpoints use a "cancel-new transaction model" with multiple phases: cancel, creation, and copy phases. This introduces latency and validation overhead.

**Limitations of legacy edit transactions:**

- Latency introduced due to multiple processing phases and persistence requirements
- Clients must chain order identifiers to track changes
- Orders always lose priority in the price queue
- Fill information is lost, making transactions difficult when trading around market price

### Atomic Amend Benefits

Atomic amends are faster than legacy edit transactions and faster than creating a new order.

**Performance advantages:**

- Reduced processing phases within the engine
- Adjustment of existing order in book vs. creation of new order
- Higher rate limits available (see spot transaction limits)
- Better race condition handling eliminates need for client-initiated cancel-new transactions
- Reduces risk, number of transactions, I/O, and latency

### API Support

Atomic amend transactions are supported across:

- **REST:** `/AmendOrder` endpoint
- **WebSockets v2:** `/amend_order` endpoint
- **WebSockets v1:** `/amendOrder` endpoint
- **FIX:** cancel-replace `35=G`

Amend history retrieval is available via REST `/OrderAmends` endpoint.

### Amending Below Fill Quantity

To keep the behaviour consistent with the client intentions (i.e. not to trade above given order quantity), amending the order quantity to below the filled quantity will be accepted.

When this occurs:

- Order quantity is amended to match the filled quantity
- Remaining order is canceled back to the client
- Standard cancel rate limit penalty applies

## Supported Fields

The atomic amend model allows modification of:

- Order limit price
- Order trigger price
- Order quantity
- Display quantity on iceberg orders

**Note:** All other order attributes are not amendable. If order type, margin characteristics, or lifetime attributes (start, end, time-in-force) need changes, cancel and resend as new order.

## Identifiers

Orders can be identified by either:

- **Kraken order identifier:** Unique over all order history, e.g., `OGYHBI-KTAUP-5TC3HO`
- **Client order identifier (`cl_ord_id`):** Generated by client, unique over open orders, can be UUID or text string, e.g., `2c6be801-1f53-4f79-a0bb-4ea1c95dfae9`

### Post Only

For changes to limit price, an additional `post_only` attribute can be added. The `post_only` flag will cause the amend to be rejected if the transaction would cause an immediate match in the book.

This prevents clients from inadvertently crossing the spread and taking liquidity.

## Queue Priority

The matching engine uses a central limit order book (CLOB) model with price-time priority basis matching.

| Field | Amend Up | Amend Down | Notes |
|-------|----------|-----------|-------|
| Order Quantity | Order placed at back of price queue | Order maintains position in price queue | — |
| Display Quantity | Displayed order maintains position in price queue, next iceberg reload at new display quantity | Displayed order maintains position in price queue | Iceberg orders only |
| Order Limit Price | Order placed at back of new price queue | Order placed at back of new price queue | — |
| Trigger Price | No impact | No impact | Trigger-based orders placed in book after triggering |

## Iceberg Order Types

Iceberg orders show display size resting in the price queue with remaining quantity hidden. When display quantity fully fills, the order appends more quantity to the back of visible price queue from hidden residual.

**Key rules:**

- Display size parameter is a maximum quantity to show in book
- Actual market quantity can be less than display size but never more
- Amending display size maintains queue priority: down maintains position, up takes effect next reload cycle
- Display size must be greater than or equal to 1/15 of order remaining size
- Validation performed on amendment of both display quantity and order quantity

## Chaining of Unacknowledged Requests

**FIX API:** Chaining of unacknowledged amends and cancels is fully supported and guaranteed to be processed in sequence by API layer and trading engine.

**WebSockets API:** The sequence of unacknowledged amends and cancels is **not** guaranteed. Sequence is non-deterministic and depends on load balancing of processes and threads.

## Caveats

Orders with conditional close terms attached are not supported by the atomic amend model.
